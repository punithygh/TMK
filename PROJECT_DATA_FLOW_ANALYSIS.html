Frontend-Backend Data Flow Analysis

Project Name: Tumakuru Connect (TMK)
Tech Stack: Next.js 15 (React 19) + Django REST Framework
Architecture: Decoupled Frontend & Backend
Analysis Date: April 22, 2026

1. Project Architecture Overview

This is a Next.js 15 (React 19) frontend application connected to a Django backend (running on port 8000). The frontend is a separate repository from the backend—they communicate via REST APIs.

Tech Stack Details

  • Frontend: Next.js 15.1.7, React 19, TypeScript, Tailwind CSS, Axios
  • Backend: Django REST Framework (DRF) on port 8000
  • Communication: HTTP/REST via JSON
  • Base URL: http://127.0.0.1:8000/api/v1
  • Deployment: Frontend can reach backend at localhost:8000 or external IPs

2. Frontend Structure & API Call Layers

2.1 Service Layer (src/services/)

This is where all backend communication originates. Each service file handles specific domain logic:

File Responsibility API Endpoints
api.ts Central Axios instance with interceptors for all HTTP requests Base: /api/v1
auth.ts Authentication operations (login, register) /auth/login/, /auth/register/
courses.ts Business listings, articles, categories, reviews, social posts /businesses/, /articles/, /categories/, /recent-reviews/, /social-posts/
user.ts User-specific operations (dashboard, bookmarks, reviews) /user/dashboard/, /businesses/{id}/bookmark/, /businesses/{id}/enquiry/, /businesses/{id}/review/, /contact/

2.2 Component Layer (src/components/)

Client-side components that make API calls for real-time, interactive user experiences:

Component API Methods Triggered By
listings-client.tsx api.get('/businesses/?...') Pagination, filtering, search queries
business-detail-client.tsx submitReview(), toggleBookmark(), submitEnquiry() User actions (review, bookmark, enquiry forms)
CategoryFilter.tsx fetchFromDjango() Category tab selection
claim-business-client.tsx Form submission (pending implementation) Business claim requests

2.3 Server Components & Pages (src/app/)

Pages that fetch data during server-side rendering (SSR) before sending HTML to browser:

Page Service Functions Called When It Fetches
page.tsx (Home) getAllCourses(), getArticles(), getCategories(), getRecentReviews(), getSocialPosts() Server render (uses Promise.all() for parallel fetching)
business/[slug]/page.tsx getOneCourse(), getAllCourses() Server render, passes data to client component
article/[slug]/page.tsx getOneArticle(), getArticles() Server render
dashboard/page.tsx getUserDashboard() Client-side after auth check
listings/page.tsx Passes params to ListingsClient Server extracts query params, client fetches

3. Complete Data Flow Examples

Example 1: User Searches for Hotels

User Types "Hotel" in Search Bar ↓ [hero.tsx] component └─ router.push('/listings?q=Hotel') ↓ Browser navigates to: /listings?q=Hotel ↓ [listings/page.tsx] (SERVER COMPONENT) ├─ Reads query params: q="Hotel" └─ Passes to ListingsClient ↓ [listings-client.tsx] (CLIENT COMPONENT) ├─ useEffect() fires on mount ├─ Builds query string: /businesses/?q=Hotel&page=1 ├─ Calls: api.get('/businesses/?q=Hotel&page=1') └─ Response contains: { results: [], count: X, next: "..." } ↓ [api.ts] INTERCEPTOR adds: ├─ Authorization: Bearer {access_token} ├─ Accept-Language: kn (or en) ├─ Trailing slash enforcement └─ CORS credentials: true ↓ HTTP Request sent to Django Backend ↓ DJANGO BACKEND (localhost:8000/api/v1/businesses/) ├─ Parses: q=Hotel parameter ├─ Queries database ├─ Filters by category/search └─ Returns JSON response ↓ [listings-client.tsx] receives data ├─ Updates state: setBusinesses(data.results) ├─ Renders: ProductCard components └─ User sees list of hotels

Example 2: User Submits a Review

User views Business Detail Page ↓ [business/[slug]/page.tsx] (SERVER) ├─ getOneCourse(slug) └─ Passes business data to BusinessDetailClient ↓ [business-detail-client.tsx] (CLIENT) ├─ Displays business info + ReviewForm ├─ User rates: 5 stars ├─ User comments: "Great place!" ├─ User clicks: "Submit Review" button └─ handleSubmit() triggers: ↓ [user.ts] submitReview(businessId, {rating: 5, comment: "..."}) ├─ api.post('/businesses/{id}/review/', data) └─ Request sent to Django ↓ DJANGO BACKEND (/businesses/{id}/review/) ├─ Creates Review record ├─ Saves to database └─ Returns: {message: "Review added"} ↓ [business-detail-client.tsx] ├─ Receives success response ├─ Sets status: "success" ├─ Shows: "Thank you! Review submitted." └─ After 3s, hides message

Example 3: User Logs In

User visits /login page ↓ [login/page.tsx] ├─ User enters: mobile, password ├─ Form submit → loginUser(mobile, password) └─ Calls from [auth.ts] ↓ [auth.ts] loginUser() ├─ api.post('/auth/login/', {username: mobile, password}) └─ Sends to Django ↓ DJANGO BACKEND (/auth/login/) ├─ Authenticates user ├─ Generates: access_token, refresh_token └─ Returns: {access, refresh, user: {...}} ↓ [AuthContext.tsx] receives response ├─ localStorage.setItem('access_token', token) ├─ Sets user state ├─ Sets isAuthenticated = true └─ Redirects to /dashboard ↓ ALL SUBSEQUENT API CALLS automatically include: └─ Authorization: Bearer {access_token} (Added by api.ts interceptor)

5. Key Infrastructure Components

5.1 Request Interception ([api.ts](api.ts))

Every API request automatically receives these modifications before being sent:

Request Interceptor adds:
  • Authorization Header: Bearer {token} (from localStorage)
  • Accept-Language: kn or en (from cookies)
  • Trailing Slash Enforcement: /businesses/ → prevents 301 redirects
  • CORS Credentials: withCredentials: true
  • Content-Type: application/json

5.2 Server vs. Client Fetch Strategy

Strategy Used By Function Features
serverFetch() Server Components (Home, Detail pages) Native fetch API - Cache: 'no-store' (always fresh)
- No interceptors
- Uses localhost:8000 (internal routing)
api (Axios) Client Components (interactive UX) Axios instance - Full interceptor chain
- Auto token injection
- Language sync
- Trailing slash enforcement

5.3 Data Caching Strategy

  • Home page: Server-renders with cache: 'no-store' → fresh data on each build
  • Listing pages: Client-side pagination + infinite scroll → dynamic filtering
  • Detail pages: Server-renders once → cached for SEO optimization
  • User dashboard: Client-side fetch after auth verification → real-time user data

5.4 Authentication Flow

Token Storage: localStorage.setItem('access_token', token)

Token Usage: Automatically attached by api.ts interceptor to all requests

Protected Routes: Dashboard checks isAuthenticated before rendering

9. Key Design Patterns & Best Practices

Pattern 1: Separation of Concerns

✓ Services handle API calls
✓ Components handle UI rendering
✓ Context manages global state
✓ Pages orchestrate data fetching

Pattern 2: Server-Side Rendering for Performance

✓ Home page: Fetches data at build time
✓ Detail pages: Generate at request time with data
✓ Client components: Handle dynamic interactions

Pattern 3: Centralized Request Interceptor

✓ All requests automatically get:
     - Bearer token for auth
     - Language preference
     - Trailing slash enforcement
✓ No duplication of these concerns in each component

Pattern 4: Dual Authentication Tokens

✓ access_token: Used in Authorization header
✓ refresh_token: Used to get new access_token when expired
✓ Stored in localStorage for persistence

Pattern 5: Bilingual Support

✓ Backend responses include both: name (English) & name_kn (Kannada)
✓ Frontend Context selects which to display based on user preference
✓ API requests send language preference in header

11. Important: No Built-in Backend

⚠️ Critical Note: This Next.js project has NO API routes. There is no /app/api/ folder. All API calls go directly to the Django backend on port 8000. The frontend is completely decoupled from the backend.

Implication: If you need server-side logic, middleware, or request transformation, it must be implemented in Django, not in Next.js.

13. Summary & Key Takeaways

Architecture Type

Decoupled Microservices Architecture

  • Frontend and Backend are separate codebases
  • Communication via REST APIs (HTTP/JSON)
  • Frontend is agnostic to backend implementation

Data Flow Summary

  • Server-side rendering: Home, detail pages → fast SEO
  • Client-side interactions: Search, filter, review → dynamic UX
  • Authentication: JWT tokens stored locally → attached to all requests
  • Internationalization: Kannada/English → server returns both, client selects

Key Files to Know

  • src/services/api.ts → All HTTP communication magic happens here
  • src/services/courses.ts → Business data fetching logic
  • src/services/user.ts → User-specific operations
  • src/context/AuthContext.tsx → Authentication & token management
  • src/app/page.tsx → Home page (server-side rendering example)

Performance Optimizations

  • Parallel fetching with Promise.all() on home page
  • Infinite scroll pagination for listings
  • Image optimization through Next.js Image component
  • Server-side rendering for fast first load
  • Caching strategies based on data freshness needs
